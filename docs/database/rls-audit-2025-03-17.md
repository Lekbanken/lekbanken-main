## Fixed Issues

### CRITICAL-1: `participants` INSERT — anyone could insert

- **Policy:** `"Service role can create participants"` on `participants`
- **Role:** `{public}` (all users, incl anonymous)
- **WITH CHECK:** `true` (no restriction)
- **Risk:** Any anonymous user could insert fake participants into any session via the Supabase REST API.
- **Code reality:** Participant creation is exclusively server-side via `POST /api/participants/sessions/join` using `createServiceRoleClient()` which bypasses RLS entirely.
- **Fix:** Dropped the policy. Service role doesn't need it, and `participants_service_insert` (authenticated/admin) remains.

### CRITICAL-2: `participant_activity_log` INSERT — anyone could insert

- **Policy:** `"Service role can insert activity logs"` on `participant_activity_log`
- **Role:** `{public}` + `WITH CHECK (true)`
- **Risk:** Same as CRITICAL-1. Any user could inject fake activity logs.
- **Code reality:** Most inserts use service_role (server API routes). Three client-side inserts from `useParticipants` hook (kick, block, role_change) use the authenticated role — these need a properly scoped policy.
- **Fix:** Dropped dangerous policy. Replaced `participant_activity_log_insert` with: `WITH CHECK (session host OR system_admin)`.

### CRITICAL-3: `session_commands` admin SELECT — user_metadata privilege escalation

- **Policy:** `session_commands_admin_select`
- **USING:** `(auth.jwt()->'user_metadata'->>'is_system_admin')::boolean = true`
- **Risk:** `user_metadata` is editable by end users via `supabase.auth.updateUser()`. Any user could self-grant admin access.
- **Fix:** Replaced with `USING (is_system_admin())` which checks `app_metadata` (not user-editable).

### HIGH-1: `participants` SELECT — all participants visible to everyone

- **Policy:** `"Participants can view themselves"` on `participants`
- **USING:** `true` (no restriction)
- **Risk:** Any user (incl anonymous) could read ALL participants across ALL sessions — display names, avatars, tokens exposed.
- **Fix:** Dropped. `participants_select_anon` already provides scoped anon access (active sessions only). Host and authenticated policies remain.

### WARN-1: 5 functions with mutable search_path

## Flagged for Review (NOT fixed)

### FLAG-1: `user_profiles_select` — `OR true` makes policy always-pass

- **USING:** `user_id = auth.uid() OR is_system_admin_jwt_only() OR true`
- **Effect:** ALL authenticated users can see ALL user profiles. The first two conditions are redundant.
- **Assessment:** Might be intentional (public profile directory for collaboration/sharing). If profiles should be public, simplify to `USING (true)`. If private, remove `OR true`.
- **Action needed:** Product decision required.

### FLAG-2: Participant heartbeat UPDATE silently fails for anonymous users

- **Context:** `useParticipantHeartbeat` calls `.update({ last_seen_at, status }).eq('participant_token', token)` using `createBrowserClient()` (anon key).
- **Problem:** No UPDATE policy on `participants` matches anonymous users. The only `{public}` UPDATE policy requires `host_user_id = auth.uid()` — always false for anon.
- **Effect:** Anonymous participant heartbeats silently update 0 rows. `last_seen_at` is never updated client-side.
- **Impact:** The system works via server-side `/api/play/me` polling, so this is non-critical, but the heartbeat feature is non-functional for anonymous participants.
- **Fix suggestion:** Add a scoped UPDATE policy for anonymous participants:
  ```sql
  CREATE POLICY "participants_heartbeat_anon" ON participants
    FOR UPDATE TO anon
    USING (EXISTS (
      SELECT 1 FROM participant_sessions ps
      WHERE ps.id = participants.session_id
        AND ps.status = 'active'
    ))
    WITH CHECK (true);
  ```
  Consider column-level restrictions to limit what anon can update.

## Policy Architecture Notes

### Duplicate policy pattern (not fixed)

Several tables have both `{public}` and `{authenticated}` policies for the same operations. Since `{public}` applies to ALL roles (including authenticated), PERMISSIVE policies are OR'd — the `{public}` policy can bypass a more restrictive `{authenticated}` policy. Tables affected: `participant_sessions`, `participants`, `plan_blocks`, `tenant_settings`, `tenants`, `user_tenant_memberships`.

This is not necessarily wrong (the `{public}` policies have proper USING checks), but creates maintenance complexity. Consider consolidating to `{authenticated}` where `auth.uid()` is required.

### Reference data SELECT policies (acceptable)

~37 tables have SELECT policies with `USING (true)` on reference/catalog data (achievements, billing plans, game artifacts, shop items, etc.). These are intentionally public read-access for catalog/lookup data.
